# 洛克王国战术 AI 助手 - 系统设计方案 (MVP 版)

## 1. 系统架构概述
本项目采用**前后端分离**的微服务架构思想。核心业务逻辑、大模型调度与数据计算统一收口在后端服务中，前端仅负责交互与展示。

* **核心后端:** Java (Spring Boot) + LangChain4j
* **本地数据库:** SQLite (极简、轻量、易打包)
* **前端 Client A:** 桌面悬浮窗 (Tauri / JavaFX)
* **前端 Client B:** QQ 机器人 (基于 OneBot V11 协议)
* **大模型驱动:** 具备 Function Calling 能力的 LLM (如 DeepSeek-V3 / 通义千问等兼容 OpenAI 格式的 API)

## 2. 数据库设计 (数据层)
数据来源：`https://wiki.lcx.cab/lk/` (通过逆向抓取前端调用的 JSON API 接口落库)。使用 SQLite 构建本地数据库。

> **⚠️ 特别注明：以下数据库表结构设计仅供参考，实际的字段定义必须严格依据最终抓取到的数据源 JSON 结构来决定并动态调整。**

### 2.1 精灵基础信息表 (`pet_info`)
*存储精灵的固有属性、被动特性和推荐配招。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 精灵唯一ID (主键) |
| `t_id` | VARCHAR(10) | Wiki 编号 (如 "001")，用于关联形态和详情页 |
| `name` | VARCHAR(50) | 精灵名称 |
| `primary_type` | VARCHAR(20) | 主属性 (如: 光, 翼) |
| `secondary_type` | VARCHAR(20) | 副属性 (可为空) |
| `hp` | INT | 精力种族值 |
| `attack` | INT | 攻击种族值 |
| `defense` | INT | 防御种族值 |
| `sp_atk` | INT | 魔攻种族值 |
| `sp_def` | INT | 魔抗种族值 |
| `speed` | INT | 速度种族值 |
| `abilities_text` | TEXT | 被动特性描述 (战术推理核心字段) |
| `description` | TEXT | 精灵简介 |
| `training_json` | TEXT | 推荐性格+配招 JSON (AI 保姆级建议的数据来源) |
| `image_url` | VARCHAR(255) | 精灵图片路径 |
| `last_updated` | DATETIME | 数据最后更新时间 |

### 2.2 精灵形态表 (`pet_form`)
*同一精灵的不同形态 (如迪莫 → 圣光迪莫)，形态拥有独立的种族值和被动特性。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 形态唯一ID (主键) |
| `pet_id` | INT | 关联的精灵ID (外键 -> `pet_info.id`) |
| `form_name` | VARCHAR(50) | 形态内部名 (如 "首领形态") |
| `form_display_name` | VARCHAR(50) | 形态显示名 (如 "圣光迪莫") |
| `hp` | INT | 该形态精力种族值 |
| `attack` | INT | 该形态攻击种族值 |
| `defense` | INT | 该形态防御种族值 |
| `sp_atk` | INT | 该形态魔攻种族值 |
| `sp_def` | INT | 该形态魔抗种族值 |
| `speed` | INT | 该形态速度种族值 |
| `abilities_text` | TEXT | 该形态的被动特性 |
| `evolution_condition` | TEXT | 进化/变形条件 |
| `attributes` | VARCHAR(50) | 该形态的属性 (可能与原始不同) |
| `form_image` | VARCHAR(255) | 形态图片路径 |

### 2.3 技能全局字典表 (`skill_info`)
*存储技能的全局属性，全库每个技能只有唯一一条记录。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 技能唯一ID (主键) |
| `name` | VARCHAR(50) | 技能名称 (如 "升龙咆哮") |
| `attribute` | VARCHAR(20) | 技能属性 (龙/虫/火/光...) |
| `category` | VARCHAR(10) | 分类: 物攻/魔攻/状态/防御 |
| `power` | VARCHAR(10) | 威力 (变化类为 "--") |
| `energy_consumption` | INT | 能量消耗 |
| `description` | TEXT | 技能效果说明 |
| `beizhu` | TEXT | 备注 (特殊触发条件等) |

### 2.4 精灵-技能关联表 (`pet_skill_mapping`)
*核心映射枢纽，记录"谁能学什么"以及技能的获取途径。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 映射唯一ID (主键) |
| `pet_id` | INT | 关联的精灵ID (外键 -> `pet_info.id`) |
| `skill_id` | INT | 关联的技能ID (外键 -> `skill_info.id`) |
| `source_type` | VARCHAR(20) | 技能来源: `innate` (精灵技能) / `skill_stone` (技能石) / `bloodline` (血脉技能) |

### 2.5 属性克制表 (`type_chart`)
*属性克制倍率关系，AI 对战推演和换宠建议的基础。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |

| `attack_type` | VARCHAR(20) | 攻击属性 |
| `defense_type` | VARCHAR(20) | 防御属性 |
| `effectiveness` | REAL | 效果倍率 (3.0 / 2.0 / 1.0 / 0.5 / 0.33) |

### 2.6 性格修正表 (`nature`)
*性格对能力值的 ±10% 修正，AI 推荐性格的数据来源。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 唯一ID (主键) |
| `name` | VARCHAR(20) | 性格名称 |
| `increased_stat` | VARCHAR(20) | 增加 10% 的属性 |
| `decreased_stat` | VARCHAR(20) | 减少 10% 的属性 |

### 2.7 推荐阵容表 (`recommended_team`) — 可选
*社区精选阵容与战术说明。*
| 字段名 | 类型 | 说明 |
| :--- | :--- | :--- |
| `id` | INT | 唯一ID (主键) |
| `team_name` | VARCHAR(100) | 阵容名称 |

## 3. 爬虫方案 (数据采集层)

数据源统一为 `https://wiki.lcx.cab/lk/`，按接口类型分为 **JSON API 直抓** 和 **HTML 页面解析** 两类。

### 3.1 JSON API 直抓 (高优先级)

| # | 目标接口 | 方法 | 写入表 | 采集策略 |
|:--|:---|:---|:---|:---|
| ① | `get_pokemon_data.php?page={N}&exclude_details=1` | GET 分页 | `pet_info` | 从 page=1 开始循环，直到返回空数组为止。需注意：`attributes` 字段为逗号分隔字符串 (如 `"光,毒"`)，需拆分为 `primary_type` / `secondary_type` |
| ② | `get_skill_data.php?page={N}&category=all&attribute=all&sort=&direction=desc&energy_value=all` | GET 分页 | `skill_info` | 同上分页循环，直接映射字段。`power` 可能为 `"--"` 需保留原值 |
| ③ | `get_pokemon_forms.php?pokemon_id={id}` | GET 逐只 | `pet_form` | 遍历 `pet_info` 表所有 `id`，逐一请求该接口。返回空数组说明无形态 |

### 3.2 HTML 页面解析 (中优先级)

| # | 目标页面 | 方法 | 写入表 | 解析要点 |
|:--|:---|:---|:---|:---|
| ④ | `detail.php?t_id={t_id}&in_iframe=1` | GET + DOM 解析 | `pet_skill_mapping` | 页面分三个区域：**精灵技能**、**可学习技能石**、**血脉技能**。每个技能的链接格式为 `skill_detail.php?id={skill_id}`，从中提取 `skill_id`，配合区域标题确定 `source_type` |
| ⑤ | `type_chart.php` | GET + 表格解析 | `type_chart` | 解析属性克制关系表格，提取攻击属性 × 防御属性 = 倍率 |
| ⑥ | `nature.php` | GET + 表格解析 | `nature` | 解析性格修正表，提取性格名 + 增加/减少的属性 |
| ⑦ | `recommended_teams.php` | GET + DOM 解析 | `recommended_team` | 解析阵容卡片，提取队名、精灵列表和战术说明 |

### 3.3 爬虫执行流程

```
步骤 1: 全量抓取精灵列表 (接口①) → 写入 pet_info
步骤 2: 全量抓取技能列表 (接口②) → 写入 skill_info
步骤 3: 逐只抓取精灵形态 (接口③) → 写入 pet_form
步骤 4: 逐只解析精灵详情 (接口④) → 写入 pet_skill_mapping
步骤 5: 一次性解析属性克制 (接口⑤) → 写入 type_chart
步骤 6: 一次性解析性格表 (接口⑥) → 写入 nature
步骤 7: 一次性解析推荐阵容 (接口⑦) → 写入 recommended_team (可选)
```

### 3.4 爬虫注意事项
* **请求频率:** 需控制请求间隔 (建议 500ms~1s)，避免给 Wiki 服务器造成压力。
* **Referer 头:** `get_pokemon_data.php` 存在请求校验，需在请求头中添加 `Referer: https://wiki.lcx.cab/lk/index.php`。
* **增量更新:** 利用 `pet_info.last_updated` 字段，后续更新时可只抓取新增/变更的精灵。
* **异常处理:** 部分精灵可能无形态数据或技能数据，需做好空值容错。
* **数据校验:** 抓取完成后，校验 `pet_skill_mapping` 中所有 `skill_id` 在 `skill_info` 中是否存在。

## 4. 核心后端架构 (Spring Boot)
后端作为整个系统的大脑，对外暴露统一的 HTTP RESTful API。

### 4.1 接口层 (Controller)
* `POST /api/v1/chat`
  * **请求体:** `{ "session_id": "用户唯一标识", "message": "用户的提问内容" }`
  * **响应体:** `{ "status": 200, "reply": "AI 生成的最终回答" }`
* `POST /api/v1/chat/stream` (SSE 流式端点)
  * **请求体:** 同上
  * **响应:** `Content-Type: text/event-stream`，逐 token 推送，支持桌面端打字机效果

### 4.2 大模型集成层 (LangChain4j)
* **System Prompt 设定:** 严格限制 AI 行为，确立"洛克王国战术大师"人设，禁止捏造游戏数值。可内嵌属性克制速查矩阵和能量系统基本规则，减少 Tool 调用开销。
* **对话记忆管理:** 使用 `MessageWindowChatMemory` 控制上下文窗口 (建议 10~20 轮)，避免超长对话导致的 token 溢出。
* **Function Calling (Tools) 注册:**

| Tool 方法签名 | 说明 | 优先级 |
| :--- | :--- | :--- |
| `@Tool getPetInfo(String petName)` | 查精灵种族值 + 被动特性 + 推荐配招 | 🔴 必须 |
| `@Tool getPetSkills(String petName)` | 查精灵可学技能 (含三种来源分类) | 🔴 必须 |
| `@Tool getTypeEffectiveness(String atkType, String defType)` | 查属性克制倍率 | 🔴 必须 |
| `@Tool getCounterPets(String typeName)` | 反查：克制某属性的精灵列表 | 🟡 重要 |
| `@Tool getPetForms(String petName)` | 查精灵所有形态及差异 | 🟡 重要 |
| `@Tool getNatureRecommendation(String petName)` | 根据种族值推荐性格 | 🟡 重要 |
| `@Tool searchBySkill(String skillName)` | 反向搜索：哪些精灵能学某技能 | 🟡 重要 |
| `@Tool getRecommendedTeams()` | 查推荐阵容列表 | 🟢 锦上添花 |

### 4.3 查询优化
* **模糊搜索:** SQLite 启用 FTS5 全文检索，对精灵名和技能名建立索引；Tool 层对查询结果为空时，自动 LIKE 模糊匹配。
* **别名支持:** `pet_info` 可扩展 `aliases` 字段存储常用别名 (如 "鸡" → "焰鸡")。

## 5. 前端接入方案

### 5.1 桌面悬浮窗端 (Desktop Overlay)
致力于提供游戏内的无缝查询体验。
* **技术选型:** Tauri + Vue/React (或 JavaFX 全栈方案)。
* **核心特性:**
  * **透明置顶:** 窗口背景透明 (`transparent: true`)，强制置顶 (`alwaysOnTop: true`)。
  * **全局快捷键:** 注册系统级热键实现快速呼出和隐藏。
  * **流式交互:** 接入后端 SSE (`/api/v1/chat/stream`)，实现打字机效果输出。

### 5.2 QQ 机器人端 (Bot Client)
致力于提供社群战术交流与快速查表功能。
* **通信协议:** OneBot V11 标准。
* **部署工具:** NapCatQQ 或 Lagrange (负责挂载 QQ 号并监听消息)。
* **交互流程:** 监听到群消息后，通过 HTTP POST 转发给 Spring Boot 后端，拿到回答后再调用发送接口推回群聊。

## 7. 开发实施路径 (MVP 阶段)
1. **数据准备:** 按第 3 节爬虫方案，逐步抓取 Wiki API 数据并写入 `data.db` (SQLite)。
2. **后端骨架:** 搭建 Spring Boot 项目，配置好 LangChain4j，生成 MyBatis/JPA 实体类与底层 SQL 查询。
3. **Tool 开发:** 按优先级依次实现 Function Calling Tools，优先完成 🔴 级别。
4. **接口调试:** 测试 `/api/v1/chat` 接口，确保多表联查和 AI 推理准确无误。
5. **接入 QQ:** 启动 NapCatQQ，完成与后端的收发对接。
6. **开发悬浮窗:** 构建桌面端 UI，联调快捷键与置顶功能，接入 SSE 流式端点。
