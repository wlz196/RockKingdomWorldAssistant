# 洛克王国世界 AI 评述系统落地方案

## 1. 目标定义

本方案的目标不是“让大模型直接给精灵写一段看起来很专业的话”，而是构建一个**可解释、可迭代、可扩展**的战术知识引擎：

1. 先将精灵、特性、技能、词条、血脉、环境数据加工成结构化战术画像。
2. 再基于画像生成评分、定位、优缺点、构筑建议。
3. 最后再由 LLM 负责自然语言表达。

面向用户定位：**进阶玩家参考构筑**。

因此系统输出需要满足：

- 结论稳定，不能同一只精灵每次评价都飘
- 能解释“为什么强 / 为什么弱”
- 能支撑后续配招、配队、环境对策扩展
- 不依赖官方环境数据，允许使用玩家整理资料逐步逼近实战结论

---

## 2. 当前方案的问题诊断

结合现有实现，当前问题主要有以下几点：

### 2.1 直接从原始资料跳到文案，缺少中间画像层

当前 `AiPetReviewService` 的做法是把以下信息直接喂给模型：

- 基础种族值
- 全局 / 本系分位
- 特性描述
- 全量技能池
- 血脉系统背景

问题在于：

- 分位信息被压缩成“顶尖 / 优秀 / 基准 / 偏弱”，信息损失严重
- 技能只是逐条罗列，没有提炼成“战术能力”
- 模型面对全技能池，无法判断实战核心构筑
- 没有环境参照，无法得出可信的竞技定位

### 2.2 技能标签体系过浅

当前技能打标只到：

- base tag：攻击 / 防御 / 状态
- 一组较粗的子标签

这不足以支持高质量评述，因为评述本质上依赖以下信息：

- 技能属于启动、压制、续航、收割、反打中的哪一类
- 触发条件是什么
- 收益给谁
- 能否与特性联动
- 能否形成自洽技能链

### 2.3 没有“常用技能组”

当前库里 `pet_common_skills` 为空，这意味着模型只能看到“能学什么”，而不是“实战常带什么”。

对于进阶玩家场景，这会显著拉低评述质量。

### 2.4 特性没有结构化建模

特性目前只有原始文本，没有提炼出：

- 常驻 / 触发 / 条件触发
- 偏输出 / 偏坦度 / 偏功能 / 偏资源
- 与什么技能类型联动
- 是补上限还是补下限

而特性往往正是精灵的真正发动机。

### 2.5 缺少环境数据层

当前系统没有：

- 主流精灵榜
- 常见阵容 archetype
- 常见速度线
- 常见威胁列表
- 主流技能组 / 血脉 / 角色定位

所以现在生成出来的内容更像“纸面分析”，不像“环境解读”。

## 3. 设计原则

### 3.1 规则 / 统计负责结论，LLM 负责表达

系统核心走“混合式方案”：

- 程序规则与数据统计负责战术画像、评分、关键结论
- LLM 只负责将结构化结论转成自然语言

这样能显著提升稳定性与可控性。

### 3.2 先做可解释，再做自动化

优先建设：

- 可人工校验的数据结构
- 可回溯的评分过程
- 可人工修订的关键字段

而不是一开始追求全自动。

### 3.3 允许人工高价值介入

尤其是以下数据，允许人工优先补充：

- 常用技能组
- 特殊词条说明
- 主流环境榜单
- 样例精灵画像

因为这些数据的价值远大于“是否完全自动生成”。

## 4. 总体架构

建议将系统拆成四层：

### 第 1 层：原始数据层

来源包括：

- `pets`
- `skill_conf_main`
- `pet_level_skills`
- `stat_quantiles_global`
- `stat_quantiles_by_type`
- 血脉 / 天赋 / 属性克制表
- 后续补充的环境资料表
- 后续补充的特性标签 / 技能标签 / 词条字典

### 第 2 层：结构化知识层

核心产物：

- 技能结构化标签
- 特性结构化标签
- 精灵常用构筑数据
- 环境威胁画像
- 精灵战术画像 JSON

### 第 3 层：评估推理层

核心产物：

- 角色定位
- 分维度评分
- 核心优势
- 核心短板
- 构筑依赖
- 环境适配度

### 第 4 层：自然语言生成层

面向前端输出：

- 一句话定位
- 进阶评述
- 核心优点
- 核心缺点
- 推荐构筑方向
- 推荐配队方向

| trigger_mode | TEXT | 常驻 / 触发 / 条件触发 |
| value_axis | TEXT | 输出 / 坦度 / 功能 / 资源 |
| trigger_tags | TEXT | JSON 数组 |
| payoff_tags | TEXT | JSON 数组 |
| synergy_tags | TEXT | JSON 数组 |

| ceiling_boost | REAL | 对上限的提升 |

| source | TEXT | manual / ai / hybrid |

## 6. 评分体系设计

建议引入 8 维评分 + 3 个总括指标。

## 6.1 主评分维度

每项 0~10 分：

1. 输出威胁
2. 生存能力
3. 速度压制
4. 功能性
5. 自洽度
6. 泛用性
7. 构筑上限
8. 操作门槛

说明：

- “操作门槛”分数越高代表越难驾驭，可在展示层转成“难度高”
- 如果不想用户看到负向分数，可在前端转译成“上手难度”

## 6.2 总括指标

1. 下限
2. 上限
3. 环境适配度

---

## 6.3 评分来源拆解

### 输出威胁
由以下因素组成：

- 主攻端基础数值分位
- 主属性与副属性打击面
- 核心输出技能数量与质量
- 条件增伤 / 连击 / 收割能力
- 特性是否能扩大输出上限

### 生存能力
由以下因素组成：

- HP / 双防分位
- 减伤 / 锁血 / 护盾 / 回复能力
- 是否容易被先手压死
- 是否有站场资源循环

### 速度压制
由以下因素组成：

- 速度全局分位 / 系内分位
- 是否有优先级技能
- 是否能靠控制 / 迅捷 / 回能间接抢节奏

### 功能性
由以下因素组成：

- 打断
- 控制
- 减益
- 驱散 / 净化
- 对策技能
- 场地 / 印记 / 转属等特殊机制

### 自洽度
由以下因素组成：

- 特性与技能是否高度联动
- 核心技能链是否闭环
- 是否能自己启动、自己维持、自己收割
- 是否严重缺组件

### 泛用性
由以下因素组成：

- 对不同队伍适配程度
- 是否强绑定特定血脉
- 是否强绑定特定队友
- 是否过度依赖特定对局条件

### 构筑上限
由以下因素组成：

- 技能池上限
- 特性滚雪球潜力
- 血脉切换收益
- 多构筑可能性

### 操作门槛
由以下因素组成：

- 是否吃对局理解
- 是否吃预判
- 是否吃技能释放顺序
- 是否容易因一步失误崩盘

## 7. 战术画像引擎设计

建议新增一个专门服务：`PetProfileEngineService`。

职责：

1. 聚合精灵基础数据
2. 聚合特性结构化标签
3. 聚合技能结构化标签
4. 聚合构筑数据
5. 聚合环境数据
6. 计算出 `pet_tactical_profiles`

建议流程如下：

### Step 1：读取基础数据

输入：

- 精灵基础属性
- 分位数表
- 主副属性
- 特性
- 技能池

产出：

- 全局百分位
- 本系百分位
- 攻防速大致档位
- 输出倾向判断

### Step 2：读取特性结构化标签

产出：

- 特性提供的价值轴
- 特性触发条件
- 特性是否影响上限 / 下限
- 特性依赖项

### Step 3：读取构筑数据

优先读取 `pet_build_profiles`：

- 核心技能
- 候选技能
- 推荐血脉
- 推荐角色标签

如果缺失，则退化为“技能池自动聚类 + 低置信度画像”。

### Step 4：技能聚合

对核心技能与候选技能做聚合，产出：

- 输出工具
- 生存工具
- 功能工具
- 资源工具
- 启动链
- 收割链
- 对策链

### Step 5：构筑依赖判定

推断：

- 是否吃技能位
- 是否吃血脉
- 是否吃队友补位
- 是否怕环境打断

### Step 6：环境对照

如果有 `meta_pet_entries`，则进一步比较：

- 是否匹配当前主流定位
- 有哪些环境天敌
- 有哪些优势对局
- 是否属于冷门高上限 / 泛用稳定 / 对策卡

### Step 7：评分与摘要输出

输出：

- 8 维评分
- 3 个总括指标
- strengths
- weaknesses
- dependencies
- role_summary
- 完整 JSON

## 8. 文案生成层设计

建议将文案生成从“一段长评”改成以下几类输出。

## 8.1 一句话定位

例：

- 中速滚雪球型多属性压制手
- 偏功能反打的中盘节奏位
- 高上限但下限偏低的爆发型副核

## 8.2 进阶评述

要求：

- 150~220 字
- 面向进阶玩家
- 语言专业但不故作玄虚
- 要明确写出优势、缺点、适用环境

## 8.3 核心优点

输出 3 条以内。

## 8.4 核心缺点

输出 3 条以内。

## 8.5 推荐构筑方向

输出：

- 推荐流派名
- 核心技能
- 可选技能
- 推荐血脉
- 构筑思路

## 8.6 配队建议

输出：

- 适合搭配什么类型队友
- 需要补什么短板
- 不适合放进什么体系

## 8.7 文案生成 Prompt 设计

LLM 不再看全量原始技能文本，而是主要看：

- 精灵战术画像 JSON
- 推荐构筑
- 环境摘要
- 目标用户：进阶玩家

示例 prompt 思路：

```text
你是洛克王国世界的高水平构筑分析师，面向进阶玩家输出战术评述。

你会收到一只精灵的结构化战术画像，请严格基于画像结论写作，不要脱离画像自由发挥。

要求：
1. 先给一句话定位。
2. 再写一段 150-220 字的进阶评述。
3. 明确写出：核心优势、核心短板、适用环境。
4. 不要写空话，不要只复述数值高低。
5. 如果画像显示“高上限低下限”，必须明确说出启动条件与翻车点。
```

这样模型会稳定很多。

## 10. 词条与隐藏机制处理方案

你提到有“迸发”“迅捷”等隐藏机制词条，这些必须系统化。

## 10.1 词条知识库

- 战术含义
- 影响的标签
- 可能影响的评分维度

例如：

### 迅捷

- formal_definition：赋予先手节奏或速度相关收益
- tactical_meaning：提升抢节奏能力，提高先手压制与收割稳定性
- related_tags：[`优先级`, `速度压制`, `节奏压制`]

### 迸发

- formal_definition：在特定状态下触发额外收益或追加伤害
- tactical_meaning：偏上限型收益，常用于爆发与滚雪球
- related_tags：[`条件增伤`, `爆发`, `滚雪球`]

## 10.2 解析策略

对于技能描述：

- 先做规则词典匹配
- 再做 LLM 补充解释
- 最后人工抽查高价值技能

即：

1. 词条识别
2. 结构化映射
3. 补标签
4. 人工修正

前端可提示：

- 高置信度：可直接参考
- 中置信度：建议结合实战调整
- 低置信度：仅供探索性参考

## 16. 迪莫示例：如何从“原始描述”变成“画像 + 评述”

你给出的参考描述是：

> 迪莫，特性是克制后给自身提供增益并回能，配合其多样的技能系别有很强的打击面，发育起来后轻松推队，缺点也比较明显，各个种族值都不太显眼，身板脆，速度不高，发育困难，并且缺乏血量续航，容易暴毙。

```json
  "feature_profile": {
    "trigger_mode": "条件触发",

  },
  "stat_profile": {
    "bulk": "低",
    "speed_tier": "偏慢",
    "raw_stats_summary": "六维整体不突出"
  },
  "movepool_profile": {
    "coverage_quality": "高",
    "sweep_potential": "高",
    "sustain_quality": "低"
  },
  "strengths": [
    "特性与多属性打击面高度联动",
    "一旦建立资源优势就容易连续压制",
    "具备较强中盘滚雪球能力"
  ],
  "weaknesses": [

  ]
}
```

### 16.2 再交给 LLM 生成进阶评述

可产出风格应接近：

> 迪莫更像一只依赖打击面与特性联动滚起节奏的中盘压制手，而不是单靠面板硬顶的泛用主核。它的价值在于克制命中后能同步拿到增益与回能，只要技能覆盖足够广，就能在对位正确时迅速拉开资源差，越打越顺。问题也很明显，迪莫六维本身并不拔尖，速度和身板都不足以保证稳定先手或强吃伤害，且缺乏可靠血量续航，一旦前几回合没能顺利启动，就很容易被高压环境直接压穿。它更适合放进能帮它创造安全发育窗口的体系，而不是单挂硬顶环境。

这就比“直接看技能池自由发挥”稳定得多。

## 19. 最终结论

最适合你当前项目的路线不是“继续优化 prompt”，而是：

**从“原始资料直出文案”切换为“结构化战术画像 -> 评分 -> 文案表达”的混合式系统。**

其中优先级最高的三件事是：

1. **补常用构筑数据**：让系统知道“实战带什么”
2. **升级技能 / 特性标签体系**：让系统知道“机制到底意味着什么”
3. **建立精灵画像层**：让系统先形成稳定结论，再让 LLM 写出来

只要这三件事做起来，评述质量会比单纯改 prompt 提升明显，而且还能自然扩展到配招、配队和环境分析。
